# 화면정의서

각 화면은 목업 제작 시 이 정의를 기준으로 삼는다. 상태(State)는 Empty / Loading / Error / Success 4종을 기본으로 정의하고, 해당 없는 경우 생략한다.

---

## S-01. 로그인

| 항목 | 내용 |
| --- | --- |
| 목적 | GitHub OAuth 로그인 진입점 |
| 진입 경로 | 최초 방문(`/`), 세션 만료 후 재방문 |
| 구성 요소 | 서비스 로고/이름, 한 줄 소개 문구, "GitHub으로 로그인" 버튼 |
| 상태 | Loading: OAuth 리다이렉트 중 버튼 비활성 |
| 인터랙션 | 버튼 클릭 → `GET /auth/github` |
| 관련 API | `GET /auth/github`, `GET /auth/github/callback` |
| 비고 | 로그인 상태로 접근 시 `/dashboard`로 즉시 리다이렉트 |

## S-03. 대시보드

| 항목 | 내용 |
| --- | --- |
| 목적 | 로그인 후 기본 진입점. 활동/회고 현황을 한눈에 파악하고 오늘의 회고 작성으로 유도 |
| 진입 경로 | 로그인 직후, GNB "대시보드" |
| 구성 요소 | ① 히트맵(최근 N주 활동), ② 스트릭 배지, ③ "오늘의 회고 작성하기" CTA 버튼, ④ 액션포인트 목록(TODO·FAILED·ROUTINE), ⑤ 회고 요약 카드 리스트(최근 순) |
| 상태 | Empty: 회고/활동 이력이 전혀 없는 신규 사용자 → 기초 가이드 + 첫 회고 작성 유도 문구. Loading: 각 위젯 스켈레톤. Error: 위젯 단위 에러 표시(전체 화면 에러 아님) |
| 상태 태그 | 회고 요약 카드에 `분석중` 태그 — `summaryStatus = ANALYZING`일 때 노출, 완료 시 요약 텍스트로 교체 |
| 인터랙션 | 히트맵 셀 클릭 → S-09 팝업, 액션포인트 항목은 이 화면에서는 읽기 전용(수정은 S-04에서만), 회고 요약 카드 클릭 → S-08 상세 |
| 관련 API | `GET /dashboard` |
| 비고 | 분석중 카드는 폴링 또는 재진입 시 재조회로 갱신 (실시간 푸시는 범위 밖) |

## S-04. 회고 작성 — 진입 (액션포인트 리뷰 / 활동 확인)

| 항목 | 내용 |
| --- | --- |
| 목적 | 회고 작성 전 기존 액션포인트를 정리하고, 오늘의 회고 소재(Git 활동 또는 직접 입력)를 확보 |
| 진입 경로 | S-03 "오늘의 회고 작성하기" 클릭 |
| 구성 요소 | ① TODO·FAILED·ROUTINE 액션포인트 목록(항목별 "완료"/"실패"/"루틴"/"철회" 토글, 전이 제한 없음, 선택 즉시 서버 반영, 목록 없으면 섹션 자체 숨김), ② Git 활동 요약(있을 경우) 또는 직접 입력 텍스트 필드(없을 경우), ③ "다음" 버튼 |
| 상태 | Loading: 활동 수집/질문 생성 대기 중 "다음" 버튼 비활성 + 로딩 인디케이터. Empty: 활동 없음 → 직접 입력 UI로 자동 전환 |
| 인터랙션 | 액션포인트 토글 선택 → 즉시 서버에 반영(절차저장형, 로컬 보관 후 일괄 반영 아님), "다음" 클릭 → 질문 생성 요청 → 성공 시 S-05로 전환 |
| 관련 API | `GET /action-points?status=TODO,FAILED,ROUTINE`, `GET /git-activities/today`, `POST /retrospects/questions/generate`. 액션포인트 리뷰 결정 저장 API는 추후 확정(FT-04 참고) |
| 비고 | 액션포인트 섹션과 활동 확인 섹션은 한 화면 내 순차 노출(액션포인트 → 활동) 구성을 기본으로 하되, 목업 단계에서 2개 스텝으로 분리할지 결정 |

## S-05. 회고 작성 — 질문 답변 (단계형 폼)

| 항목 | 내용 |
| --- | --- |
| 목적 | 질문에 하나씩 답변하며 K/P/T를 실시간으로 누적 |
| 진입 경로 | S-04에서 질문 생성 완료 |
| 구성 요소 | ① 진행률 표시(예: "1/3"), ② 현재 질문 텍스트, ③ 답변 입력 필드(서술형), ④ "다음" 버튼, ⑤ (선택) 지금까지 누적된 K/P/T 미리보기 사이드 영역 |
| 상태 | Loading: 답변 제출 후 KPT 재구성 대기 중 "다음" 버튼 비활성. Error: 재구성 실패 시 해당 질문에 인라인 에러(전체 리다이렉트 아님 — 단, 재시도 실패가 반복되면 공통 에러 정책 적용) |
| 인터랙션 | 답변 입력 후 "다음" → `POST /retrospects/kpt/reconstruct` → 성공 시 다음 질문 또는(마지막 질문이면) S-06으로 전환 |
| 관련 API | `POST /retrospects/kpt/reconstruct` |
| 비고 | 챗봇형 대화 UI가 아니라 회원가입 마법사형 단일 질문 폼임을 목업에서 명확히 구분할 것 |

## S-06. 회고 작성 — 최종 확인 (KPT 확정 → 액션포인트 작성)

| 항목 | 내용 |
| --- | --- |
| 목적 | ① 누적된 K/P/T 전체를 검토·수정 후 확정 저장, ② 이어서 액션포인트를 작성(후보 채택/직접 추가)하고 목적(Purpose)을 연결 |
| 진입 경로 | S-05 마지막 질문 답변 완료(① 진입), KPT 저장 완료(② 진입, 같은 화면 내 전환) |
| 구성 요소 | **①KPT 확정**: Keep/Problem/Try 목록(편집 가능, Try 최대 6개), 항목 추가 버튼(K/P/T 각각), "저장" 버튼. **②액션포인트 작성**(① 저장 후 노출): 액션포인트 후보 목록(진입 시 Try 기반 AI 후보 자동 생성 + S-04에서 "실패"/"루틴"으로 결정한 항목을 참고한 파생 초안 포함 — 채택/수정/삭제는 저장 전까지 자유, 채택 시 즉시 서버 저장. S-04에서 리뷰한 기존 액션포인트 원본은 여기서 수정하지 않음), 액션포인트 직접 추가 버튼(추가 즉시 서버 저장), 목적(Purpose) 선택/입력 UI(항목별로 기존 활성 목적 드롭다운 선택 또는 새 목적 입력, 태그로 표시 + 태그 수정 버튼), "완료"/"건너뛰기" 버튼 |
| 상태 | Loading: ① 저장 요청 중 버튼 비활성. ② 액션포인트 후보 생성 대기 중(진입 시 자동), 후보 채택·직접 추가·목적 저장 등 개별 요청 중 해당 항목 인라인 로딩 |
| 인터랙션 | ① Keep/Problem/Try 인라인 수정/삭제, "+"로 항목 추가, "저장" 클릭 → KPT 확정 저장(스트릭 +1, 비동기 요약/인사이트/제목 분석 job 적재) → 성공 시 ②로 전환. ② 후보 채택 또는 직접 추가 → 즉시 서버 저장(TODO 상태 생성), 목적 드롭다운 선택/새 목적 입력 → 즉시 서버 저장, 목적 태그 수정 → "잦은 목적 수정은 방향성을 해칠 수 있어요" 안내 노출 후 저장(같은 목적을 공유하는 다른 액션포인트 표시도 함께 갱신), "완료"/"건너뛰기" 클릭 → 별도 저장 없이 S-03 복귀 |
| 관련 API | `POST /retrospects/action-points/generate` (② 진입 시 후보 생성, 저장 없음). KPT 확정 저장 / 액션포인트 개별 저장 / 목적 저장 API는 추후 확정(FT-07 참고) |
| 비고 | 활성(TODO+FAILED+ROUTINE) 액션포인트가 16개인 상태에서 새 항목을 추가하면 그 항목만 저장이 생략되고 경고 문구가 표시됨(다른 항목 추가는 계속 가능). 목적 드롭다운 후보는 현재 활성 액션포인트에 걸린 목적만 노출. 목적은 별도 관리 화면 없이 이 화면에서만 추가·수정 가능. 액션포인트는 생성 후 수정 불가·중복 체크 없음·존댓말/표준어 보정 없이 입력 그대로 저장. ①을 벗어나면(뒤로가기 등) 미저장 KPT 내용은 소실 — 확인 다이얼로그 노출 여부는 목업 단계 결정 사항 |

| 항목 | 내용 |
| --- | --- |
| 목적 | 개별 회고의 전체 내용(질문/답변, K/P/T, AI 요약·인사이트) 열람 및 삭제 |
| 진입 경로 | S-07 카드 클릭, S-09 팝업 내 링크, S-03 요약 카드 클릭 |

| 비고 | 수정 버튼 없음 (회고는 수정 불가 원칙) |

## 공통 에러 표시

전 화면 공통으로 별도 "에러 화면"을 두지 않는다. API 실패 시 토스트/얼럿 형태로 에러 메시지를 노출한 뒤 `/`로 리다이렉션한다 ([[03-user-flow#5-공통-에러-정책]] 참고). 구체적 UI 컴포넌트는 목업 단계에서 확정한다.
